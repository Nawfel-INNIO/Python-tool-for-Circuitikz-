    Canvas Architecture — scene.py

    This document explains how the visual circuit editor canvas is implemented.

    Two Classes

    The canvas is built with Qt's Graphics View Framework using two classes:

    Class Base Class Role
    CircuitScene QGraphicsScene Owns the data, handles all logic (placement, selection, drag, draw)
    CircuitCanvas QGraphicsView Thin viewport — zoom, mouse tracking, key forwarding

    CircuitCanvas delegates almost everything to CircuitScene. It only handles: - Mouse wheel → zoom (scale by 1.15×) - Mouse move → forwards to scene.update_preview() or scene.drag_move() - Escape / Delete keys - zoom_to_fit() — computes bounding rect of all components and calls fitInView()

    All the real work happens in CircuitScene.

    Grid System

    • Grid size: 50 px per unit (GRID_SIZE = 50)
    • Grid extent: ±15 units → 31×31 grid = 961 dot positions
    • Grid dots are drawn once in _init_grid() as tiny ellipses at z-level -10
    • All component coordinates are stored in grid units (integers), not pixels
    • Pixel conversion: px = gx * GRID_SIZE, py = gy * GRID_SIZE

    Snapping

    _snap(pos) converts a raw mouse position to the nearest grid point: 1. Round to nearest integer grid coordinates 2. If in placement or wiring mode, also check all pin offsets of existing components 3. If a pin is within 1.5 grid units, snap to that pin instead

    State Machine (Modes)

    The scene operates as a state machine with these modes:

    idle ──────────────┬──► place_single ──► idle     (one-click: grounds, nodes)
                   ├──► place_first ──► place_second ──► idle   (two-click: R, C, L, etc.)
                   ├──► busbar_first ──► busbar_second ──► idle
                   └──► wire ──► wire ──► ... ──► idle   (multi-click, double-click to finish)
    • idle: click to select, right-click to deselect, drag selected component
    • place_single: one click places a node-style component (ground, VCC, transistor, op-amp)
    • place_first / place_second: two clicks define start and end of a two-terminal component
    • busbar_first / busbar_second: two clicks define a busbar span
    • wire: each click adds a waypoint; consecutive pairs become short components; double-click or Escape finishes

    Mode transitions happen via _set_mode(), which also fires the on_mode_changed callback so MainWindow can update the status bar text.

    Mouse Handling

    mousePressEvent (left click)

    Depending on current mode: - idle: try to start a drag on the selected component, or call _try_select() to find the nearest component - place_single: add component, return to idle - place_first: store first position, move to place_second - place_second: add two-terminal component between first and second click, return to idle - wire: append waypoint, create a short segment between last two waypoints

    mousePressEvent (right click)

    Cancels the current placement mode or deselects.

    mouseDoubleClickEvent

    In wire mode, finishes the wire and returns to idle.

    mouseMoveEvent

    • If dragging: calls drag_move() to reposition the drag target
    • Otherwise: calls update_preview() to show a crosshair at the snapped position and a dashed line from the first click point

    Selection

    _try_select(gx, gy) finds the closest component or busbar: - Node-style components: distance = squared distance to (x1, y1) - Two-terminal components: distance = point-to-segment distance (using projection formula in _point_to_segment_dist2) - Busbars: same point-to-segment distance - Selects the nearest object if distance ≤ 2.5 grid units; otherwise deselects

    Selection fires on_selection_changed so the property panel updates.

    Dragging

    When clicking on an already-selected component in idle mode:

    1. _drag_hit() checks what part was clicked:
    2. Near (x1, y1) → drag "start" endpoint
    3. Near (x2, y2) → drag "end" endpoint
    4. Near the body (line segment) → drag "body" (moves both endpoints)
    5. Node-style components → always drag as "body"
    6. drag_move() updates the component coordinates to the new snapped position on each mouse move
    7. mouseReleaseEvent ends the drag

    Busbars have similar logic via _busbar_drag_hit() (start/end/body).

    Drawing (_redraw)

    Every time the circuit changes, _redraw() is called:

    1. Removes all items in _dynamic_items from the scene
    2. Redraws all busbars (_draw_busbar)
    3. Redraws all components (_draw_comp)
    4. Draws connection node dots (_draw_nodes)
    5. Fires on_circuit_changed callback (triggers LaTeX regeneration)

    Component Drawing (_draw_comp)

    Components are drawn differently based on their type:

    Category Visual Representation
    short (wire) Simple line in black/orange
    Two-terminal (R, C, L, etc.) Line with a labeled box at midpoint
    Ground variants Vertical line + horizontal bars (decreasing width)
    VCC / VEE Vertical line + horizontal bar
    Transistors Rectangle with type name
    Op-amps Triangle with +/− labels
    Logic gates, ICs Labeled rectangle (size varies by category)
    Transformers Two side-by-side circles (coils)
    Tubes Circle with label
    Generator / Motor Circle with G or M letter

    Colors: - Selected → orange (255, 140, 0) - Custom color → from comp.color - Default → blue (0, 60, 180) - Wires → black when unselected, orange when selected

    Annotations on Two-Terminal Components

    Three annotation layers are drawn alongside the component line:

    1. Voltage (red): value text + polarity symbols (+/−), positioned perpendicular to the line. Direction controlled by voltage_dir (v=, v<=, v^=, v_=).
    2. Current (green): arrow + label, positioned below or above. Direction controlled by current_dir (i=, i<=, i^=, i_=).
    3. Flow (blue): arrow + label, similar to current. Direction controlled by flow_dir (f=, f<=, f^=, f_=).

    Node Dots (_draw_nodes)

    Black filled dots (6px diameter, z-level 4) at every endpoint and pin position, indicating connection points.

    Pin Offsets

    Multi-port components (transistors, op-amps, ICs) define pin_offsets in the COMPONENTS registry — relative grid offsets for each named pin. These are drawn as small gray dots with labels, and the snapping system uses them for precise wiring.

    Preview System

    update_preview() draws temporary visual aids (stored in _preview_items, cleared each frame): - Orange dashed crosshair at the current snapped position - Green highlight circle when snapped to a pin - Dashed line from first click point (during two-click placement) - Dashed line from last wire waypoint (during wire mode)

    Undo / Redo

    • _snapshot() serializes the entire Circuit to a dict (via circuit_to_dict) and pushes it onto _undo_stack. Clears _redo_stack.
    • undo() pushes current state to redo stack, pops from undo stack, deserializes
    • redo() reverses the process
    • Snapshots are taken before any mutation (place, delete, drag start, property change)

    Z-Level Ordering

    Z-Level Content
    -10 Grid dots
    1 Lines (wires, busbars, component lines)
    2 Shapes (rectangles, ellipses, polygons, label backgrounds)
    3 Text labels, annotations, pin labels
    4 Connection node dots
    5 Pin snap highlights
